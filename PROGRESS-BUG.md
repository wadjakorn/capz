## Open

- [ ] **Settings wiped after updater-driven update** — after the in-app updater applies a new version, user-modified settings (hotkeys, output dir, filename template, capture options, etc.) reset to defaults. Likely the store file path / scope changes between bundle versions, or the new install ships a fresh default store that overwrites the existing one. Investigate `tauri-plugin-store` file location (`$APPCONFIG/...`), whether the updater replaces it, and whether bundle resource files shadow the user store on first launch. Fix: persist the store outside the app bundle / resources dir, never bundle a default `store.json` that overwrites on install, and add a migration path for store schema bumps so old keys survive across versions. Verify by: set non-default settings → run updater → confirm values intact post-relaunch. Affects macOS + Windows.
  - **Status (2026-05-28):** path hardening + schema-version scaffold + startup diagnostic log landed on branch `claude/infallible-dijkstra-ec55fb`. Store now uses absolute `app_data_dir()/config.json` via `services::config_store::config_store_path()`; all 7 Rust call sites switched. Frontend adds `schemaVersion=1` and `migrateConfig()` (identity for v1). Local repro test inconclusive — `config.json` survived bundle replace byte-identical, but UI still showed defaults; two `capz.app` bundles on disk (Applications + build path) confounded LaunchServices. **Defer verification to next released version**: cut release → install → set custom settings → trigger in-app updater → confirm persistence. If still wipes, diagnose by inspecting `config.json` content shape vs `store.get("app")` key path — suspect key wrapping or shape mismatch, not store file location.
- [x] **macOS Screen Recording permission recovery flow** — landed on branch `claude/practical-chatterjee-bec16d`. Rewrote `InertGrantRecoveryDialog` ([src/components/onboarding/InertGrantRecoveryDialog.tsx](src/components/onboarding/InertGrantRecoveryDialog.tsx)) as a 4-step machine (Remove → Re-prompt → Toggle on → Relaunch) with explicit delete-first copy, step badges, probe status, and auto-advance polling — detects when the user toggles the new TCC row on in System Settings and bumps to step 4 without needing a dialog button click. Relaunch button always enabled with helper text for users who want to skip ahead. Card layout uses flex column + `max-h-full` + scrolling body so content can't bleed past the editor window's bottom edge (replaces a flaky `100vh` calc). Onboarding Permission step gained an `inert` state ([src/components/onboarding/OnboardingView.tsx](src/components/onboarding/OnboardingView.tsx)) that runs a real capture probe and routes to the dialog when preflight=true but capture=blank. Added proactive `useScreenRecordingHealthCheck` ([src/lib/notice.ts](src/lib/notice.ts)) that fires a persistent "Screen Recording is not working" toast on app launch + window focus whenever capture is broken, with a `Fix permission…` action that opens the dialog directly. Stripped redundant toast from `useStalePermissionAfterUpdateListener` (marker write retained). Cross-ref Developer ID signing item below (a stable cdhash mostly retires this flow for future updates).
- [ ] **Settings randomly fires "Saved" toast** — spurious toast not tied to a user save action. Investigate store-init / hydration paths in [src/app/settings/page.tsx](src/app/settings/page.tsx) and any `useEffect` that writes to `useSettings` on mount.
- [ ] **macOS Developer ID signing (upstream fix for stale TCC)** — switch [tauri.conf.json `bundle.macOS.signingIdentity`](src-tauri/tauri.conf.json) from `"-"` (ad-hoc) to a Developer ID Application certificate so the cdhash is stable across releases and the existing TCC grant survives updates. Needs paid Apple Developer Program account, cert + private key in keychain, notarization step in the release flow, and CI secret wiring (cert.p12 + password + ASC API key) — distinct from the Ed25519 updater key. Once landed, the detection-and-prompt above will mostly idle (still useful as a safety net for users on the last ad-hoc build).
- [ ] **macOS "Capture Window" includes menu bar** — `xcap` window enumeration on macOS exposes the system menu bar as a capturable window. Filter it out (and likely Dock, Wallpaper, status items) in the window picker so users can't select non-app surfaces.
- [ ] **Windows allows multiple app instances** — on Windows, launching capz while it's already running spawns a second instance instead of focusing the existing one. Should be single-instance: use [tauri-plugin-single-instance](https://v2.tauri.app/plugin/single-instance/) to detect duplicate launch, focus the existing editor window (or tray), and exit the new process. Wire `app.handle().plugin(tauri_plugin_single_instance::init(...))` with a callback that calls `windows::show_editor` on the existing instance. macOS already single-instance by default via LSUIElement / bundle ID; plugin still safe to register cross-platform.
- [ ] **Windows "Capture Window" mode unusable — no window clickable** — on Windows the window-picker overlay shows the dim backdrop + hover hint but clicking any visible window does nothing; no highlight appears, no capture fires. Suspect causes: (a) `list_capture_windows` returning empty or filtering everything out on Windows (`current_monitor()` / `is_minimized()` / `pid` checks behave differently vs macOS, or `w.id()` errors on Win32 HWNDs), (b) coordinate-space mismatch — overlay hit-test uses logical CSS pixels while `WindowOverlayInfo` x/y/w/h are physical pixels on Windows (overlay also positioned in physical pixels per [windows.rs:84-88](src-tauri/src/windows.rs:84)) so rects land outside the React viewport, (c) z-order / focus issue where overlay never becomes `active` so click handler bails. First step: log `list_capture_windows` result length + first item coords on Windows; verify hit-test rect vs cursor coord space.
- [ ] **Continue-able tools require two clicks after first placement** — with the numbered-pin tool, after placing a pin (which becomes auto-selected), the next click on empty canvas only deselects instead of placing the next pin. Cause: `handleMouseDown` in [src/components/editor/EditorStage.tsx:499](src/components/editor/EditorStage.tsx:499) early-returns whenever `empty && hadSelection`, even for the pin tool which is already exempted from the tool-reset side of the same branch. Fix: detect continue-able tools (currently only `pin`) and let the click fall through to the pin-placement path — `add()` already overwrites `selectedId` so the new pin becomes selected automatically. Verify by placing 3+ pins in 3 consecutive single clicks; numbers must advance 1,2,3.
- [ ] **Onboarding "Granted" sticks after mid-session revoke** — `CGPreflightScreenCaptureAccess()` caches the grant for the lifetime of the process, so toggling capz off in System Settings while the app is running still reports `true`. Polling preflight will never flip back to denied. Need an actual capture probe (e.g. `CGWindowListCreateImage` 1×1 or `CGDisplayStream`) and inspect for null/black pixels to detect revocation. The `app:permission-revoked` toast already fires when a real capture fails — this is only the onboarding StatusCard that lies.
